# The `improve-codebase-architecture` Skill

## Definition

`improve-codebase-architecture` is a **Matt Pocock skill** (GitHub: `mattpocock/skills`) that scans a codebase to identify shallow module candidates — clusters of small, tightly-coupled files that should be consolidated into deep modules with simple interfaces. It is demonstrated live in the "Advanced AI Coding Techniques" workshop (2026).

**Skill invocation:** `/improve-codebase-architecture` in Claude Code

## What It Does

The skill performs an architectural scan:

1. **Module detection** — identifies clusters of files that depend on each other
2. **Depth analysis** — evaluates whether each cluster has a simple or complex interface
3. **Shallow candidate identification** — flags clusters with many small files, high coupling, complex interfaces
4. **Consolidation recommendations** — proposes how to redesign the interface to hide the complexity
5. **Dependency substitution suggestions** — finds where in-memory/test doubles can replace real dependencies (e.g., swap SQLite for an in-memory DB in tests)

### Example Output (from workshop)

```
Architectural Improvement Candidates:

1. quiz-scoring-service
   - files: quiz-scoring.ts, reorder.ts, validate.ts, ...
   - current interface: 12 exported functions
   - proposed interface: 1 function: scoreQuiz(quiz, answers)
   - test gap: ZERO TESTS (biggest gap)
   - dependency category: local, substitutable with in-memory double
   - reason: tightly coupled cluster, single responsibility violated

2. video-editor infrastructure
   - files: frontend-editor.ts, backend-api.ts, transport.ts
   - current: crosses frontend/backend boundary
   - proposed: single deep module wrapping entire editor flow
   - benefit: AI can see entire flow, test entire flow
   - result: night-and-day improvement in AI ability to make changes
```

## The Refactoring Process

### Step 1: Identify Shallow Module Clusters

Look for:
- Many small files (5-50 lines each)
- High inter-folder dependencies
- Similar naming patterns (often auto-generated by AI)
- Zero or minimal test coverage

### Step 2: Find the Natural Seam

The seam is where behavior can be changed without editing the code in place. Mark the seam with a deep interface:
- 1-3 functions maximum
- All complexity hidden behind the interface

### Step 3: Consolidate Behind the Interface

Move implementation into the deep module. The callers now depend only on the interface, not the internals.

### Step 4: Add Boundary Tests

Replace N shallow tests with 1-3 boundary tests that verify behavior at the interface.

### Step 5: Delete the Old Files

If the deletion test passes (complexity stays hidden), delete the old shallow files.

## The Video Editor Example

The most compelling example from the workshop: Pocock's video editor (a browser-based video editor spanning frontend to backend).

**Problem:** AI couldn't make changes effectively — too many files, too many cross-boundary dependencies.

**Solution:** Wrapped the entire front-end-to-back-end flow in a single deep module using a discriminated union type between frontend and backend types.

**Result:**

> "It meant that AI could see the entire flow, could act on the entire flow, and test on the entire flow. And honestly, it was just night and day in terms of the ability of AI to actually make changes."

## Connection to Deep Modules

This skill is the **operationalization** of John Ousterhout's deep module theory within an AI coding workflow:

| Ousterhout Concept | `improve-codebase-architecture` Practice |
|-------------------|------------------------------------------|
| Deep module | Single cluster consolidated behind simple interface |
| Simple interface | 1-3 function boundary |
| Information hiding | Internal files hidden from callers |
| Deletion test | Deleting the module would collapse callers |

## The "Zero Tests" Discovery

One of the skill's most valuable outputs is identifying modules with zero test coverage. These are the highest-risk modules for AI iteration — they are exactly where SlopCodeBench degradation starts.

> "Quiz scoring service currently has zero test. This is the biggest gap."

Run `improve-codebase-architecture` on any codebase where AI productivity is low. The output will show you exactly where to deepen the modules, which modules have zero tests, and which consolidation patterns will give the biggest return in AI navigation improvement.
